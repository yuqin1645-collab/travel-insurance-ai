## 一、当前问题概述

### 1.1 核心问题

`ai_review_result` 表当前承载 **87 个字段**，航班延误与行李延误共用同一张宽表，导致：

- **险种私有字段污染主表**：行李延误的 `baggage_*` 字段对航班延误案件永远为 NULL，反之亦然
- **未来扩展失控**：每新增一个险种（如"随身财产损失"、"旅行取消"），主表就会新增 20+ 字段
- **部分行李字段代码已提取但未建列**：`baggage_receipt_time`、`baggage_delay_hours` 等在 `_extract_review_fields` 中已提取写入，但**不在任何 migration 中**，实际写入会报错
- **死字段泛滥**：`payout_basis`、`supplementary_deadline`、`frontend_response`、`metadata` 从未被任何写入路径赋值

### 1.2 缺失字段清单（代码已引用但表中不存在）

## 二、拆分原则

### 2.1 字段归属规则

| 归属 | 规则 | 举例 |
|------|------|------|
| **主表** | 任意险种都必须有的字段 | 审核结论、赔付金额、身份校验 |
| **航班延误子表** | 仅航班延误险使用的字段 | 计划起飞时间、飞常准数据、改签信息 |
| **行李延误子表** | 仅行李延误险使用的字段 | 行李签收时间、PIR编号、行李延误档位 |

### 2.2 新增险种时的影响

| 范围 | 是否需要改主表 |
|------|----------------|
| 新增险种（如"旅行取消"） | **不需要改主表**，只需新建 `ai_travel_cancellation_data` 子表 |
| 新增公用字段（如 `ai_model_version`） | 需要改主表 |

## 五、数据迁移策略

### 5.1 迁移步骤

```
Step 1: 新建 ai_flight_delay_data 表
        ↓
Step 2: 新建 ai_baggage_delay_data 表
        ↓
Step 3: ai_review_result 新增缺失列 + AI追溯列
        ↓
Step 4: 从 ai_review_result 迁移航班字段 → ai_flight_delay_data
        （仅迁移 claim_type = 'flight_delay' 的记录）
        ↓
Step 5: 从 ai_review_result 迁移行李字段 → ai_baggage_delay_data
        （仅迁移 claim_type = 'baggage_delay' 的记录）
        ↓
Step 6: 从 ai_review_result 删除已迁移的列
        ↓
Step 7: 更新视图、DAO、模型
```

### 5.2 安全约束

- 所有操作在事务中执行
- 迁移前后保持 `forceid` 唯一性
- 先写子表再删主表字段，确保数据不丢失
- 迁移前备份 `ai_review_result` 全表

CREATE OR REPLACE VIEW v_claim_audit_summary AS
SELECT

    r.applicant_name,

### 6.2 v_audit_statistics

```sql
CREATE OR REPLACE VIEW v_audit_statistics AS
SELECT
    DATE(r.audit_time) AS audit_date,
    r.claim_type,
    COUNT(*) AS total_count,
    SUM(r.audit_result = '通过') AS pass_count,
    SUM(r.audit_result = '拒绝') AS reject_count,
    SUM(r.is_additional = 'Y') AS supplementary_count,
    AVG(f.delay_duration_minutes) AS avg_delay_minutes,
    SUM(r.payout_amount) AS total_payout
FROM ai_review_result r
LEFT JOIN ai_flight_delay_data f ON r.forceid = f.forceid
WHERE r.audit_time IS NOT NULL
GROUP BY DATE(r.audit_time), r.claim_type;
```

> 新增 `claim_type` 维度分组，可按险种分别统计。

## 九、后续扩展指导

### 9.1 新增第三险种（如"随身财产损失"）

只需三步：

1. 新建子表 `ai_baggage_property_data`（字段仅行李财产损失相关）
2. `claim_type` 枚举新增 `baggage_property` 值
3. `_extract_review_fields` 新增 `property_fields` 分支

**不需要改动主表**（除非有全新类型的公用字段）。

### 9.2 主表新增字段的原则

只有满足以下条件之一才在主表新增：

- 两个以上险种都需要此字段
- 领导驾驶舱需要直接统计此指标
- AI训练必须此字段作为模型输入/输出
