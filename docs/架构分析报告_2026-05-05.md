## 二、Pipeline 拆分合规性分析

### 2.1 规范要求（CLAUDE.md 第7节）

> `pipeline.py` 文件行数**不得超过 500 行**。超过时必须拆分出 `stages/` 子目录。
> `pipeline.py` 只做编排，不做实现。

### 2.2 实际测量

| 模块 | pipeline.py 行数 | 超限 | 超限比例 | stages/ 目录 | 合规？ |
|------|-----------------|------|----------|-------------|--------|
| `baggage_delay` | **523 行** | +23 行 | 4.6% | ✅ 有（3文件, 538行） | ⚠️ 轻微超限 |
| `flight_delay` | **729 行** | +229 行 | 45.8% | ✅ 有（7文件, 2,153行） | ❌ 严重超限 |
| `baggage_damage` | **119 行** | — | — | ❌ 无（扁平文件结构） | ✅ 行数合规 |

### 2.3 详细分析

#### baggage_delay/pipeline.py（523行）

**超限原因**：视觉识别结果合并逻辑（第112-246行，约135行）和材料门禁逻辑（第330-392行，约63行）直接写在 pipeline 中，未下沉到 stages。

**问题代码段**：
- 第112-246行：`vision_extract` → `ai_parsed` 的字段合并（含 PIR 二次提取、交叉校验、低置信度清除等），约135行纯业务逻辑
- 第330-392行：材料门禁的 `_has_flag()` 内联函数 + 逐项检查逻辑，约63行
- 第394-406行：人工复核触发关键词检测，约13行

**建议**：将视觉合并逻辑提取到 `stages/handlers.py` 的 `_merge_vision_results()`，将材料门禁逻辑提取到 `stages/handlers.py` 的 `_run_material_gate_detailed()`。

#### flight_delay/pipeline.py（729行）

**超限原因**：这是三个模块中最严重的违规。大量编排逻辑和数据合并逻辑直接写在 pipeline 中：

- 第138-329行（约192行）：**stage1.2 Vision 合并** — 将视觉抽取结果合并到 parsed 结构，包含 claim_focus、schedule_revision_chain、flight、route、alternate、evidence 等7个子块的逐字段合并
- 第331-468行（约138行）：**stage1.3 飞常准查询** — 候选航班构建、多候选查询循环、联程场景处理
- 第469-608行（约140行）：**stage1.4 接驳/替代航班查询** — 联程改签航班查询、数据覆盖逻辑
- 第681-710行（约30行）：**硬校验标记拼接** — 11种硬校验结果的 Remark 拼接

**核心问题**：这些代码段都是"编排+实现混合体"，既做数据传递又做业务判断。按照规范，它们应该下沉到 `stages/` 中。

#### baggage_damage/pipeline.py（119行）

**唯一完全合规的模块**。使用 `AuditPipeline` 框架，pipeline 只做：
1. 前置校验（`run_precheck`）
2. 材料提取（`MaterialExtractor`）
3. 出行提示（`record_travel_vs_effective_hint`）
4. 构建 `PipelineContext` + `AuditPipeline` 并执行

但 baggage_damage 采用了**完全不同的架构模式**（AuditPipeline + StageHandler），与另外两个模块的 StageRunner 模式不一致。

## 五、代码质量分析

### 5.1 异常处理

**问题严重程度：高**

全项目共发现 **40+ 处裸 `except Exception`**，分布在：

| 文件 | 数量 | 典型模式 |
|------|------|---------|
| `flight_delay/stages/hardcheck.py` | 12处 | `except Exception:` 静默吞掉 |
| `baggage_delay/stages/utils.py` | 4处 | 解析函数中静默返回 None |
| `flight_delay/stages/utils.py` | 5处 | 解析函数中静默返回 |
| `baggage_damage/extractors.py` | 5处 | 抽取失败静默降级 |
| `baggage_delay/pipeline.py` | 3处 | 视觉/航班查询失败降级 |
| `flight_delay/pipeline.py` | 5处 | 查询失败降级 |

**典型问题代码**：

```python
# baggage_delay/stages/utils.py:20
except Exception:       # ← 裸异常，丢失错误信息
    return None

# flight_delay/stages/hardcheck.py:71
except Exception:       # ← 日期解析失败静默跳过
    continue
```

**建议**：
- 至少记录 `exc_info=True` 日志
- 区分预期异常（如解析失败）和意外异常（如系统错误）
- 关键路径（赔付计算）不应静默吞异常

### 5.2 硬编码

| 硬编码类型 | 位置 | 风险 |
|-----------|------|------|
| 赔付档位（500/1000/1500） | `calculator.py:69-74` | 与规则文件不同步风险 |
| 赔付档位（300/600/900/1200） | `payout.py` | 同上 |
| 起赔门槛（6h/5h） | 多处 | 与条款不同步风险 |
| 关键词列表 | `pipeline.py` 内联定义 | 不便维护 |
| 安联顺延15天 | `policy_validity.py:139` | 硬编码业务规则 |
| 飞常准 API 错误分类关键词 | `utils.py:120-125` | 与 API 变更不同步 |

### 5.3 类型安全

- 使用了 `from __future__ import annotations`（现代化）
- 大量使用 `Dict[str, Any]` 和 `Optional[str]`，类型信息丢失严重
- `RuleResult` 使用 `@dataclass`，设计良好
- `ClaimModule` 使用 `Protocol`，设计良好
- 但 pipeline 函数签名中 `reviewer: Any` 完全丢失类型信息

### 5.4 提示词系统一致性

| 险种 | 提示词文件数 | 使用 `{{include:}}` | 共享块引用 |
|------|------------|---------------------|-----------|
| `baggage_delay` | 4 | ✅ 是（3处引用） | policy_validity, war_exclusion, flight_info_extract |
| `flight_delay` | 3 | ❌ 否 | 无（规则直接写在提示词中） |
| `baggage_damage` | 7 | ❌ 否 | 无 |

**问题**：flight_delay 的 `02_audit_decision.txt`（180行）包含了大量规则描述，这些内容本应通过 `{{include:}}` 引用共享块。baggage_damage 同样如此。

## 八、总结

旅行险 AI 审核系统在**架构设计理念**上是优秀的——四层分层、模块隔离、规则知识库、共享提示词块等设计都体现了良好的工程素养。但在**执行层面**存在明显的技术债务：

1. **最严重问题**：flight_delay/pipeline.py 729行（超限45.8%），大量编排+实现混合代码
2. **最普遍问题**：工具函数重复定义（9处 `_parse_date`），缺乏统一基础库
3. **最隐蔽问题**：三个模块三种架构风格，长期维护成本高
4. **最易修复问题**：baggage_delay 仅超限23行，1小时内可完成拆分

建议优先解决 P0 问题（pipeline 拆分），然后逐步推进 P1/P2 改进，使项目在保持架构理念先进性的同时，实现层面也达到同等水准。
