我们领导感觉这个随身财产比较复杂，我们已经在了一周多了但是现在的进度十分的缓慢，所以想着先做一些简单的模块，这个先放一下，但是我们现在就面临的一个最大的问题就是，我们现在的代码，架构是否支持，我们在原有的基础上再新加一个模块，

其次是条款责任。@static/旅行险条款 这个也是。不会加载错误
然后是我认为最终要的就是咱们的代码。@app/claim_ai_reviewer.py 这个代码我可以说是完全服务于随身财产的里面的逻辑都是关于随身财产的。所以我们是新建一个py文件用来处理航安延误的案件呢。或者你有其他更好的方案吗。注意：我们的有10多种类型的案件，我觉得我们要是创建10多个py的话可能不是会很好管理。
最主要是架构设计时一定要考虑到，我在改bug时你可以很好的定位到这个bug所在的文件位置和具体的那个函数。@app/claim_ai_reviewer.py 就但这一个代码就2000多行，到时候业务越来越多累加的代码也会越来越多，你到时候如果出现了bug我们怎么能快速地定位到这个bug并解决这个也是你在架构设计的时候需要考虑的一个点。
现在按照我上面考虑的点，帮我分析一下，我们现在的系统架构需要怎么重新设计才能很好的做到文件管理，prompts管理、代码迭代时出现bug修复等问题。

结论先说清楚
你们现在的系统可以在原有基础上扩展到 10+ 业务类型，但前提是要把当前“随身财产一体化脚本”拆成可插拔的业务模块（plugin）+ 通用引擎（workflow）。否则代码、prompts、条款都会越来越难维护，Bug 也很难定位。

目标架构（你关心的 4 件事：文件管理、prompts隔离、条款隔离、Bug定位）
建议采用“产品线模块 + 注册表”的结构：

通用层（不含任何业务规则）
负责：加载案件、附件预处理、OCR/vision、OpenRouter/Gemini 调用、重试、日志、结果落库、DebugInfo、统一输出 schema
业务模块层（每种案件类型一个 module，但不是一个巨大的 py）
每个业务模块包含：该类型的 prompts、条款选择、材料清单、关键字段硬抽取、阶段编排（workflow/stages）
这样你新增“航班延误”只需要新增一个模块目录，不会改动随身财产的 prompts/条款/逻辑，也不会误加载。

目录与隔离设计（推荐落地形态）
1) Prompts 管理：按“案件类型命名空间”隔离
把 @prompts 从“平铺文件”改为：

prompts/
common/（跨产品通用：输出格式约束、通用提示片段）
baggage_damage/（随身财产-承运人损坏等）
01_coverage_check.txt
02_material_evidence_vision.txt
02_material_check_evidence.txt
03_accident_judgment.txt
04_compensation_calculation.txt
05_final_summary.txt
flight_delay/
同样一套 stage prompts（名字可以一致，但在不同目录下）
并改 prompt_loader 支持 namespace 参数：

prompt_loader.format("02_material_check_evidence", namespace="baggage_damage", ...)
这样航班延误永远不可能加载到随身财产 prompt（除非 namespace 写错，而这可以用强校验避免）。
2) 条款管理：按“案件类型/产品代码”注册与强校验
把 @static/旅行险条款 配成一个 registry（映射表）：

static/旅行险条款/
baggage_damage/个人随身物品保险条款.txt
liability_damage/旅行个人责任(物损)保险条款.txt
……
代码层做一个 PolicyTermsRegistry：

输入：claim_type / BenefitName / 未来你们的“服务手册目录名称”
输出：条款文件路径
强约束：当前你们全部是随身财产，则 registry 只允许返回 个人随身物品保险条款.txt；如果有人误配成“旅行个人责任(物损)”，直接 raise，让问题暴露在启动阶段，而不是默默跑错。
3) 代码管理：拆 app/claim_ai_reviewer.py 成“引擎 + 模块”
你担心 2000+ 行继续膨胀，这是必然的。建议拆成：

app/engine/
workflow.py：状态机/阶段重试/StageRunner（你们现在的 per-stage retry 继续放这里）
types.py：统一结果 schema（Remark/IsAdditional/KeyConclusions/DebugInfo）
errors.py：统一异常兜底返回体（系统异常一律转人工）
stage_fallbacks.py：单阶段失败兜底返回体（某一阶段 err 时统一转人工，主流程只调用构造器）
precheck.py：快速预检查（有效期拒赔/重复理赔跳过/金额字段预处理）
travel_hint.py：出行日期提示（仅记录供参考，不自动拒赔）
pipeline_labels.py：阶段文案/编号集中管理（避免主流程硬编码）
pipeline_log.py：阶段日志输出工具（统一格式）
app/logging_utils.py：统一 logger、上下文字段（forceid/stage/attempt）
app/modules/
baggage_damage/
module.py：实现一个 ClaimModule 接口（见下）
extractors.py：原价/第三方赔付/日期等硬抽取都集中在这里（便于定位和单测）
materials.py：材料缺件门禁/缺关键材料早退
accident.py：事故判责结果的除外责任早退（命中除外直接拒赔）
coverage.py：系统异常判定与转人工输出（避免误判拒赔）
compensation.py：核算早退/无可赔付金额口径输出
decision.py：拒赔返回体统一构造
final.py：审核通过返回体统一构造
prompts.py：模块内 prompt 名称常量/映射（防拼错，后续可补）
flight_delay/（未来新增）
app/main.py 或 app/router.py
ModuleRegistry：根据案件类型选择模块
如果当前只有随身财产：registry 固定返回 baggage_damage
关键点：不是“10 个巨大 py”，而是“10 个模块目录，每个模块内多个小文件”，每个文件职责单一，Bug 很好定位。

业务模块接口（保证可扩展 + 易定位）
定义一个非常小的接口（概念上）：

ClaimModule
name
supported_claim_types
policy_terms_path(claim_info) -> Path
prompt_namespace -> str
material_gate(...)（材料门禁：缺购买凭证就补件等）
run_stages(...) -> ReviewResult（阶段编排）
这样“航班延误”就是新增一个模块实现同样接口，不会动随身财产代码。

Bug 快速定位：你要的“改 bug 能马上找到文件/函数”
建议强制做到这几条（架构层面保障）：

每个阶段一个函数/文件：materials/coverage/accident/compensation/final 分别在模块内独立文件
每个硬抽取一个函数：extract_purchase_amount(), extract_third_party_paid()… 且写 extraction_debug 指向“规则名/命中片段/文件名”
日志必须带 module：log extra 增加 module=baggage_damage，以后看到一条日志就知道归哪个模块
结果 DebugInfo 带 stage_trace：每个 stage 的输入摘要/输出摘要/异常原因（你们已经有 DebugInfo，继续强化即可）
迁移策略（不耽误你们现在进度）
不建议一次性重写。推荐最小风险两步走：

第 1 步（1-2 天）：先做“路由/注册表”
强制所有案件走 baggage_damage module
prompts/条款加 namespace/registry，但先只注册随身财产
第 2 步（持续）：把 claim_ai_reviewer.py 里的业务逻辑逐块搬进 module 目录
优先搬：抽取器、材料门禁、除外责任规则（这些最容易出错、最需要单独维护）
回答你对条款的强调（你刚刚那条要求）
在上面这套设计里，当前所有案件都属于随身财产时，registry 会让系统只能加载static/旅行险条款/个人随身物品保险条款.txt，任何试图加载 旅行个人责任(物损)保险条款.txt 都会直接报错阻止运行——从机制上杜绝“加载错条款”。
