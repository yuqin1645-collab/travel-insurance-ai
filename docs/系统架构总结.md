# 旅行险 AI 审核系统架构文档

## 1. 文档目的

本文档定义旅行险 AI 审核系统的正式架构基线，用于统一以下事项：

- 代码分层与模块边界
- Prompt 与条款的命名空间隔离
- 案件类型扩展方式
- 审核链路可观测性与故障定位规范

## 2. 设计目标

系统架构需满足以下核心目标：

- **可扩展**：新增案件类型应以“新增模块 + 注册”完成，避免修改既有险种核心逻辑。
- **强隔离**：不同 `claim_type` 的 Prompt、条款、结果目录必须隔离，禁止隐式回退到其他险种资源。
- **可观测**：每个阶段必须可追踪（`forceid/stage/attempt/module`），支持快速定位异常。
- **可治理**：系统异常与证据不足要可区分，并映射到明确动作（人工复核/补件）。

## 3. 总体架构

系统采用“入口编排层 + 通用引擎层 + 业务模块层 + 能力层”四层结构。

### 3.1 入口编排层

主要文件：`app/claim_ai_reviewer.py`、调度器与生产工作流。

职责：

- 读取案件与附件上下文
- 识别 `claim_type` 并路由到目标模块
- 统一调用模块 pipeline
- 落盘审核结果与调度状态

约束：

- 入口层不得新增险种业务细节判断
- 业务规则应下沉到 `app/modules/<claim_type>/`

### 3.2 通用引擎层

目录：`app/engine/`

职责：

- 阶段执行与重试：`workflow.py`（`StageRunner`）
- 阶段异常兜底：`stage_fallbacks.py`
- 通用预检查与日志工具：`precheck.py`、`pipeline_log.py` 等

约束：

- 引擎层不承载险种专属口径

### 3.3 业务模块层

目录：`app/modules/`

每个险种一个独立目录，典型结构：

- `module.py`：模块元信息（`claim_type`、`prompt_namespace`、`policy_terms_path`）
- `pipeline.py`：该险种主审核流程
- 其他按职责拆分文件（抽取、材料、责任、赔付、结论）

当前已启用模块：

- `baggage_damage`（随身财产）
- `flight_delay`（航班延误）
- `baggage_delay`（行李延误）

### 3.4 能力层（Skills）

目录：`app/skills/`

提供跨模块复用能力，例如：

- 航班官方数据查询：`flight_lookup.py`
- 赔付计算、机场/天气/战争因素查询等

定位：

- 模块决定业务口径
- Skills 提供可复用事实能力

## 4. 关键隔离机制

### 4.1 Prompt 隔离

目录约束：

- `prompts/<claim_type>/...`
- 调用必须显式携带 `namespace`

调用示例：

`prompt_loader.format("01_data_parse_and_timezone", namespace="baggage_delay", ...)`

### 4.2 条款隔离

目录约束：

- `static/旅行险条款/<claim_type>/...`

条款选择统一通过 `PolicyTermsRegistry`：

- 输入：`claim_type`
- 输出：条款文件路径
- 未配置映射：直接抛错，不允许静默降级

### 4.3 模块路由隔离

通过 `ModuleRegistry` 注册模块，路由逻辑基于 `claim_type` 分发到对应 pipeline。

约束：

- 禁止跨模块加载 Prompt/条款
- 结果落盘目录按 `claim_type` 隔离

## 5. 审核结果与数据契约

统一输出结构：

- `Remark`
- `IsAdditional`
- `KeyConclusions`
- `DebugInfo`

其中：

- `IsAdditional = "Y"` 表示需补件或人工复核
- `DebugInfo` 必须保留阶段轨迹、关键中间结果与异常摘要

## 6. 行李延误模块当前实现（2026-04）

`baggage_delay` 已进入可运行状态，当前实现要点如下：

1. **独立模块与命名空间**
   - `app/modules/baggage_delay/`
   - `prompts/baggage_delay/`
   - `static/旅行险条款/baggage_delay/行李延误保险条款.txt`

2. **时长核算口径**
   - 优先：`行李签收时间（最晚） - 首次航班实际到达时间`
   - 回退：结构化 `delay_hours`，再回退文本小时数

3. **官方航班数据优先**
   - 接入 `flight_lookup` 获取 `actual_arr`
   - 成功时覆盖时长起算点来源

4. **失败分类可审计**
   - 官方查询失败分为：
     - `system_error`（系统异常 -> 人工复核）
     - `evidence_gap`（证据不足 -> 补件）
   - 分类结果写入 `KeyConclusions` 与 `DebugInfo`

5. **赔付规则**
   - `<6h` 拒赔
   - `6~12h` 500 元
   - `12~18h` 1000 元
   - `>=18h` 1500 元
   - 并受索赔金额、保额、剩余保额约束

## 7. 新增案件类型标准流程

新增任意 `claim_type` 时，必须按以下步骤执行：

1. 新建 `app/modules/<claim_type>/`
2. 实现 `module.py` 与 `pipeline.py`
3. 新建 `prompts/<claim_type>/`
4. 新建 `static/旅行险条款/<claim_type>/`
5. 在 `ModuleRegistry` 注册模块
6. 在 `PolicyTermsRegistry` 注册条款映射
7. 在入口路由中增加识别逻辑

## 8. 可观测性与故障定位规范

日志最小字段要求：

- `forceid`
- `stage`
- `attempt`
- `module`

排查建议：

- 材料问题：优先看模块 `materials`/门禁逻辑
- 时长问题：优先看模块 `pipeline` 中核算函数
- 赔付问题：优先看模块 `compensation`/核算逻辑
- 资源加载问题：优先看 `ModuleRegistry`、`PolicyTermsRegistry`、`prompt_loader`

## 9. 架构治理原则

- 不允许在入口层继续堆叠险种规则
- 不允许跨命名空间复用 Prompt 与条款
- 不允许将“系统异常”和“证据不足”混为同一结论
- 每次新增险种必须同步更新本架构文档
