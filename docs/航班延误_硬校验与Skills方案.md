## 航班延误：代码侧硬校验 + MCP Skills 方案

本文档用于把《航班延误审核规则.txt》落地到系统实现层，明确：
- 哪些规则适合做**代码侧硬校验**（确定性门禁/后处理，不依赖模型发挥）
- 哪些规则需要模型做**材料理解/抽取/解释**
- 需要接入哪些外部数据（航班权威数据、机场国家/时区、战争/冲突、天气预警等）
- 建议封装为哪些 **MCP 服务 / Skills**，供模型调用
- 分阶段实施流程表（一步一步做，直到方案完成）

---

## 1. 定义：什么是“代码侧硬校验”

### 1.1 硬校验（建议落代码）
满足以下任一特征的规则，优先落为代码逻辑（确定性）：
- **可判定**：输出是明确的 True/False/Unknown，不依赖语义猜测
- **可重复**：同样输入一定得到同样结论（可回归测试）
- **可审计**：能记录“命中哪条规则/依据是什么/缺什么数据”
- **关键合规**：规则明确写了“不可直接拒赔/必须转人工/必须补材”这类口径

常见形态：
- 门禁（Gate）：不通过直接拒赔/补材/转人工
- 计算（Calc）：延误时长、阈值比较、取长原则
- 比对（Compare）：材料 vs 官方数据一致性
- 时间线（Timeline）：投保/订票/改签 vs 预警/公告发布时间

### 1.2 模型处理（适合交给模型）
满足以下特征的规则，模型更擅长：
- 需要从**图片/PDF/OCR**中抽取字段
- 需要语义理解、关联多份材料（登机牌/行程单/改签记录/延误证明）
- 输出是“补材清单”“矛盾点说明”“证据链叙述”等自然语言内容

一句话：**模型负责“看与说”，代码负责“算与判”。**

## 2. 规则拆分：硬校验 vs 模型处理（对齐《航班延误审核规则》）

> 说明：下面每条都给出建议的落地点（stage0/后处理/模型prompt/人工复核）。

### 2.1 前置准入审核（不通过直接拒赔）——硬校验优先

- **主险合同状态/权益有效性**
  - 硬校验：从保单/合同接口或结构化字段判断（有效/终止/失效）
  - 缺数据：转人工/补材（不要让模型猜）

- **权益有效期（含安联顺延规则）**
  - 硬校验：日期逻辑最适合落代码
  - 需要数据：
    - 原保单生效/满期（已在 claim_info 示例存在）
    - 实际出境日期/第一次出境日期（若系统无该字段，需要从材料抽取或新增接口）
    - 延误发生时间（至少能落到 planned/alt 的某个时间点）
  - 输出口径：
    - 明确超期：拒赔（并记录规则与关键日期）
    - 关键日期缺失：补材/转人工

- **权益人身份校验（姓名/证件号一致性）**
  - 硬校验：claim_info vs 材料抽取字段的比对（或 claim_info vs 保单接口返回）
  - 缺委托/继承等说明材料：转人工或拒赔（按你们业务口径）

### 2.2 基础赔付条件前置校验——硬校验优先，模型负责抽取

- **航班属性：客运民航班机**
  - 硬校验优先：航班权威数据源若能返回航班性质/承运信息则直接判定
  - 模型辅助：材料中出现“货运/包机”等字样时抽取线索
  - 无法核验：转人工/补材

- **延误触发原因：必须是外部原因**
  - 建议做成“**原因枚举白名单**”的硬校验：
    - 外部原因集合：恶劣天气/自然灾害/机械故障/罢工/航空管制/超售/其他条款列明外部原因
  - 模型负责：从延误证明/通知短信等抽取原因文本并映射到枚举
  - 无法核验原因：补材（延误证明）或转人工

### 2.3 核心赔付条件：延误时长“取长原则”——必须硬校验

- **取长原则**
  - 硬校验：从结构化时间点计算：
    - a) planned_dep -> alt_dep（最早便利替代起飞）
    - b) planned_arr -> alt_arr（替代抵达原目的地）
    - 取 max(a, b)
  - 模型负责：抽取 planned/alt 的时间点与来源证据

- **阈值判断（2小时/4小时等）**
  - 硬校验：阈值来自条款结构化字段或条款解析结果；缺条款时可用默认兜底，但必须打低置信度并建议人工复核

- **异常处理：时间缺失/矛盾**
  - 硬校验策略：
    - 缺关键时间点：补材/转人工（不可硬算）
    - 时间矛盾：转人工（记录矛盾字段与来源）

### 2.4 免责情形——规则口径明确的部分做硬校验

- **无书面延误证明**
  - 规则：不能直接拒赔，先查询第三方，查不到再补材
  - 硬校验：缺证明/证明缺关键字段 => 强制输出“需补齐资料”+给出可查询建议

- **未按时登乘/票号未使用**
  - 模型负责：从登机牌/客票号材料抽取
  - 硬校验：若拿到“票号使用状态/登机记录”，可确定性判定；否则转人工

- **未搭乘最早便利替代航班**
  - 模型负责：从承运人改签记录/登机牌抽取“最早便利”证据
  - 硬校验：一旦存在可核验的替代航班列表，可比对“安排 vs 实乘”并确定性判定

### 2.5 联程/中转限制（中转接驳延误、境内中转）——硬校验 + 机场国家能力

- **中转接驳延误免责**
  - 模型负责：识别语义“前序延误导致误后续接驳”，抽取航段关系
  - 硬校验：若能构建航段链路（段1/段2），可确定性判定“只赔符合阈值的段、接驳段免责”

- **中转地在境内拒赔**
  - 需要：机场 IATA -> 国家（CN）能力（见 Skill：airport.resolve_country）
  - 硬校验：一旦中转地国家=CN，直接命中免责（并记录中转地IATA/国家）

### 2.6 反欺诈：可预见因素时间线——硬校验（需要外部数据源）

- **投保/订票/改签 vs 台风预警/罢工公告发布时间**
  - 硬校验：对关键时间点做比较（发生在预警后且行为异常 => 提示欺诈/拒赔/转人工）
  - 需要外部数据：
    - 气象预警发布时间（权威气象数据）
    - 罢工公告发布时间（事件/新闻源，或先走人工维护表）
  - 输出建议：
    - 初期阶段先做“强提示 + 转人工”，避免误伤

### 2.7 战争因素拒赔（你们的新增要求）——建议“维护表 + 外部证据更新”

- 业务口径：战争因素需要拒赔；当前考虑通过 Gemini 或 Security Link 获取信息
- 建议落地方式（更稳健）：
  1) **战争/冲突维护表**（可配置）：国家/地区 + 生效期（类似你们文档示例：阿联酋 20260226-20260630）
  2) 代码硬校验：命中维护表 => exclusion_triggered=true（拒赔或转人工按口径）
  3) 外部数据源用于“维护表更新/校验证据”，由技能服务提供证据摘要

### 2.8 材料完整性校验——模型抽取 + 硬校验门禁

- **权益补偿给付申请书**
  - 模型：抽取申请人信息、延误详情、索赔金额等字段
  - 硬校验：必填字段缺失 => 补材/转人工（记录缺失字段清单）

- **会员权益救援服务卡/保险凭证**
  - 模型：OCR识别凭证信息
  - 硬校验：与保单信息比对，校验凭证有效性

- **申请人身份证明**
  - 模型：OCR抽取证件信息（姓名、证件号）
  - 硬校验：与保单权益人信息比对，不一致 => 拒赔/转人工

- **承运人延误证明**
  - 模型：抽取航班号、延误起止时间、延误原因、出具单位
  - 硬校验：核心字段缺失 => 补材；与官方数据不一致 => 标记欺诈

- **辅助材料场景**
  - 委托代办：模型抽取委托书信息，硬校验完整性（安联当前不涉及）
  - 遗产继承：模型抽取继承文件，硬校验合法性
  - 监护人场景：模型抽取监护关系，硬校验完整性

### 2.9 赔付金额审核——必须硬校验

- **金额计算规则**
  - 硬校验：根据条款结构化字段（delay_hours → amount 映射表）计算
  - 公式：amount = tier_lookup(delay_hours, policy_tier_config)

- **索赔金额校验**
  - 硬校验：
    - if claim_amount > calculated_amount: 按 calculated_amount 赔付
    - if claim_amount < calculated_amount: 按 claim_amount 赔付
    - 记录差异原因

### 2.10 服务区域校验——硬校验

- **承保区域校验**
  - 硬校验：获取保单承保区域列表，校验延误发生地是否在范围内
  - 命中规则：超出范围 => 拒赔 + 标记人工复核

### 2.11 复核机制与人工介入点

- **未决件（AI无法出结论）**
  - 材料信息模糊
  - 时间数据矛盾
  - 辅助材料缺失但不影响核心判定
  - 疑似欺诈但无法确认
  - → 标记"人工复核"，推送人工审核岗

- **已决件（AI已判责）**
  - 审核通过案件 → 人工复核确认后推送赔付
  - 审核拒赔案件 → 人工复核确认后执行拒赔
  - 人工有权修正AI结论

## 3. 外部数据/接口清单（实现硬校验所必需）

### 3.1 航班权威数据（用于时长计算、材料真实性比对、原因枚举）
目标字段（最小集）：
- 航班状态：起飞/到达/取消/备降
- planned_dep/planned_arr、actual_dep/actual_arr
- 延误原因（标准枚举/原因文本）
- 执行航班号/承运人（用于 codeshare）

候选数据源：
- 飞常准/航旅纵横/航司官方（以你们现有接入能力为准）

### 3.2 机场/国家/时区数据（用于境内中转、服务区域、UTC换算）
目标能力：
- IATA -> 国家（ISO2/ISO3）/城市/时区(IANA)

候选数据源：
- 可用公共机场数据库（离线更稳）或第三方 airports API

### 3.3 战争/冲突数据（用于战争因素拒赔与证据）
建议采用“维护表 + 外部证据”组合。

候选数据源（可做证据与自动化辅助）：
- ACLED API（冲突事件级数据，需账号/Token）
- UCDP API（研究级冲突数据，需 access token）
- ReliefWeb API v2（公开，需 appname 参数；可作为报告/危机证据）

### 3.4 气象/公告数据（用于可预见因素时间线）
目标字段：
- 预警发布时间、影响区域、预警等级、有效期

数据源建议：
- 先接权威气象渠道（国内可优先中央气象台），罢工公告可先用人工维护表/白名单，逐步自动化

## 4. MCP 服务 / Skills 设计（供模型调用）

> 目标：让模型能“取证据/取权威数据”，但关键判定仍由硬校验完成。

### Skill A：flight.lookup_status（航班权威查询）
- 输入（示例）：
  - flight_no、date、dep_iata（可选）、arr_iata（可选）
- 输出（示例）：
  - planned_dep/arr、actual_dep/arr、status、delay_reason_code、delay_reason_text、operating_flight_no、operating_carrier、source、queried_at

### Skill B：airport.resolve_country（机场三字码解析）
- 输入：iata
- 输出：country_code、country_name、timezone(IANA)、is_domestic_cn

### Skill C：war.check_country_risk（战争/冲突风险查询 + 证据）
- 输入：country_code、date（或日期范围）
- 输出：
  - is_war_risk=true/false/unknown
  - evidence[]：来自 ACLED/UCDP/ReliefWeb 的条目摘要（标题/日期/来源ID）
  - suggestion：reject/manual_review

### Skill D：weather.lookup_alerts（气象预警查询，可选）
- 输入：airport/city、date range
- 输出：alert_level、published_at、valid_from/to、summary

### Skill E：policy.lookup_effective_window（保单权益窗口/顺延所需字段）
- 输入：PolicyNo、被保险人信息
- 输出：effective_from/to、insurer、是否安联、是否允许顺延、出境日期（若系统可拿）

### Skill F：booking.lookup_ticket_status（客票号使用状态查询）
- 输入：ticket_no、flight_no、date
- 输出：status（已使用/未使用/已退票）、used_flight_no、used_date

### Skill G：flight.lookup_rebooking（改签记录查询）
- 输入：original_flight_no、date、passenger_name
- 输出：rebooking_records[]（原航班、改签后航班、改签时间、改签原因）

### Skill H：policy.lookup_coverage_area（保单承保区域查询）
- 输入：PolicyNo
- 输出：coverage_regions[]（国家/地区列表）、region_type（全球/区域/单国）

### Skill I：material.verify_evidence（材料真实性校验）
- 输入：material_type、material_path、expected_values
- 输出：is_authentic（true/false）、confidence、suspicion_markers[]、evidence_comparison

## 6. 落地约束与降级策略（必须写清）

- 外部 API 不可用/超时：
  - **不得硬拒赔**；应降级为“需补齐资料/转人工”，并记录缺口（DebugInfo）
- 数据缺失：
  - 关键字段缺失（时间点/国家/原因）：补材/转人工
- 规则冲突：
  - 以“权益有效性/免责”优先，且保留人工复核入口
