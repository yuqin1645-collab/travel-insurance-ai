# ADR-001：模块化与命名空间隔离

## 状态

已采纳（Accepted）

## 日期

2026-04-13

## 背景

旅行险 AI 审核系统在持续扩展险种时，出现以下典型风险：

- 入口文件业务逻辑持续膨胀，导致维护成本和改动风险上升。
- Prompt 与条款文件存在误加载风险（跨险种串用）。
- 新增险种时无法做到“低影响扩展”，容易改坏存量流程。
- 审核异常缺少统一分流口径（系统异常 vs 证据不足）。

为支持后续多险种并行演进，需要建立可扩展、强隔离、可观测的架构基线。

## 决策

采用“入口编排层 + 通用引擎层 + 业务模块层 + 能力层”的四层架构，并以 `claim_type` 为核心命名空间进行全链路隔离。

### 1. 模块化决策

- 每个险种一个独立模块目录：`app/modules/<claim_type>/`
- 入口层仅负责路由，不承载险种规则
- 通过 `ModuleRegistry` 实现模块注册和分发

### 2. Prompt 隔离决策

- Prompt 按险种目录隔离：`prompts/<claim_type>/`
- 所有 prompt 调用必须显式传 `namespace`

### 3. 条款隔离决策

- 条款按险种目录隔离：`static/旅行险条款/<claim_type>/`
- 统一通过 `PolicyTermsRegistry` 映射
- 未配置映射时直接报错，禁止静默回退到其他险种条款

### 4. 结果与可观测性决策

- 输出结构统一：`Remark / IsAdditional / KeyConclusions / DebugInfo`
- 日志上下文统一要求：`forceid/stage/attempt/module`
- 审核失败统一区分：
  - 证据不足 -> 补件
  - 系统异常 -> 人工复核

## 落地范围（本次）

### 已落地项

1. 新增 `baggage_delay` 模块并注册到 `ModuleRegistry`
2. 新增 `baggage_delay` 条款目录并接入 `PolicyTermsRegistry`
3. 新增 `prompts/baggage_delay/` 独立命名空间
4. `claim_type` 路由支持 `baggage_delay`
5. 行李延误规则实现（阶段化）：
   - 材料门禁
   - 事故类型筛选
   - 时长门槛（6小时）
   - 阶梯赔付（500/1000/1500）
6. 时长核算升级为规则口径优先：
   - 最晚签收时间 - 首次航班实际到达时间
   - 官方航班数据优先（`flight_lookup`）
7. 官方查询失败分类：
   - `system_error` -> 人工复核
   - `evidence_gap` -> 补件

## 备选方案与取舍

### 方案 A：继续在单体入口文件中扩展

- 优点：短期改动快
- 缺点：长期复杂度指数上升，难以并行演进与定位问题

### 方案 B：按险种拆模块，但不做命名空间隔离

- 优点：代码结构表面更清晰
- 缺点：Prompt/条款误加载风险仍然存在

### 方案 C（采纳）：模块化 + 全链路命名空间隔离

- 优点：
  - 扩展可预测
  - 资源加载可控
  - 观测与排障路径清晰
- 代价：
  - 初期有注册与规范成本
  - 需要持续维护模块边界与文档

## 影响评估

### 正向影响

- 新增险种的改动范围可控
- 误加载 Prompt/条款风险显著降低
- 故障定位可直接落到模块与阶段
- 可逐步推进自动化回归与治理

### 负向影响

- 初期需要维护更多目录与注册项
- 文档、代码、配置需保持同步

## 约束与执行规则

1. 不允许在入口层新增险种专属审核规则。
2. 不允许跨 `claim_type` 读取 Prompt 和条款。
3. 每新增一个 `claim_type`，必须同时更新：
   - 模块注册
   - 条款注册
   - Prompt 命名空间
   - 架构文档与 ADR
4. 结论必须区分“补件”与“人工复核”触发原因。

## 验收标准

- [x] 新增 `claim_type` 时，存在独立目录 `app/modules/<claim_type>/`
- [x] 新增 `claim_type` 时，存在独立 Prompt 命名空间 `prompts/<claim_type>/`
- [x] 新增 `claim_type` 时，存在独立条款目录 `static/旅行险条款/<claim_type>/`
- [x] `ModuleRegistry` 已注册对应模块
- [x] `PolicyTermsRegistry` 已注册对应条款映射
- [x] 审核结果落盘目录按 `claim_type` 隔离
- [x] `DebugInfo` 中可见阶段执行与关键失败原因
- [x] 对官方航班查询失败已区分 `system_error` 与 `evidence_gap`

## 回滚与退出策略

### 回滚条件

- 新模块导致生产审核显著失败率上升
- 发现无法接受的误判（拒赔/补件/通过路径大面积偏差）
- 命名空间隔离改动导致历史案件无法正常审核

### 回滚步骤

1. 将路由中对应 `claim_type` 暂时回切到稳定模块（例如回退到既有审核链路）。
2. 在 `ModuleRegistry` 中禁用有问题模块映射。
3. 在 `PolicyTermsRegistry` 保留映射但暂停入口调用，避免误加载。
4. 保留失败样本与日志，进入缺陷修复分支后再灰度恢复。

### 数据与兼容处理

- 结果文件不删除，按 `claim_type` 目录保留用于审计。
- 回滚期间新增案件继续可处理，历史结果不重写。
- 回滚完成后通过样例回归集验证后再恢复新模块流量。
