### ⚠️ 需关注

**1. 异常吞噬（hardcheck.py 多处）**

```python
# hardcheck.py:564-565, 571-572, 592-593, 607-608
except Exception:
    pass
```

**风险：** 多处 `except Exception: pass` 完全静默吞掉异常，一旦内部逻辑出错（如 KeyError、TypeError），不会有任何日志，排查困难。

**建议：** 至少记录 debug 日志：
```python
except Exception as e:
    result["debug_notes"].append(f"xxx校验降级: {e}")
```

## 三、逻辑错误与边界条件

### 🔴 严重

**1. baggage_delay/pipeline.py — `_check_info_consistency` 重复调用（L258, L438）**

同一函数在流程中被调用两次，第一次在 L258（前置校验阶段），第二次在 L438（赔付核算前）。如果第一次已通过，第二次不可能失败，属于**冗余调用**。更严重的是，如果中间 `ai_parsed` 被修改导致两次结果不一致，会产生逻辑矛盾。

**建议：** 删除 L438 的重复调用，或将第一次结果缓存复用。

**2. baggage_delay/pipeline.py — `delay_calc` 重复计算（L409-413）**

```python
delay_calc = _compute_delay_hours_by_rule(ai_parsed or {}, text_blob)  # L409
delay_hours = delay_calc.get("delay_hours")
if debug.get("transfer_flight_receipt", {}).get("receipt_time_set"):
    delay_calc["receipt_time_source"] = "transfer_flight_arrival"
    delay_calc = _compute_delay_hours_by_rule(ai_parsed or {}, text_blob)  # L413 重复!
    delay_hours = delay_calc.get("delay_hours")
```

当转运航班签收时间被设置后，`_compute_delay_hours_by_rule` 被调用了两次。第一次的结果被丢弃，造成不必要的计算开销。且 L409 的 `delay_hours` 在 L414 被覆盖，L409 的调用完全无意义。

**建议：** 将 L409 的计算移到条件分支内，避免无条件计算后丢弃。

### 🟡 中等

**3. delay_calc.py — 联程改签判定条件脆弱（L237-246）**

```python
connecting_rebooking_suspicion = (
    is_connecting
    and (
        is_conn_rebooking
        or (
            isinstance(c, int) and isinstance(d, int) and d > 0
            and c > d * 1.2  # 魔数 1.2
        )
    )
)
```

`c > d * 1.2` 这个启发式条件缺乏注释说明为什么是 1.2 倍。如果起飞延误恰好是到达延误的 1.19 倍，判定结果会不同，边界不稳定。

**建议：** 添加注释说明 1.2 的业务含义，或改用更明确的判定条件。

**4. hardcheck.py — `vision_extract` 参数默认值 None（L166）**

```python
def _run_hardcheck(
    ...
    vision_extract: Dict[str, Any] = None,  # 类型标注为 Dict，默认值为 None
) -> Dict[str, Any]:
```

类型标注与默认值不一致。虽然代码中多处使用 `(vision_extract or {}).get(...)` 做了防御，但 L288 处直接访问 `(vision_extract or {}).get("evidence", {}).get("exit_datetime")`，链式访问在 `evidence` 不是 dict 时会抛异常。

**建议：** 统一使用 `Optional[Dict[str, Any]] = None`，并在函数开头做标准化处理。

**5. handlers.py — `_try_transfer_flight_receipt_time` 修改入参（L283）**

```python
ai_parsed["baggage_receipt_time"] = arr_str  # 直接修改传入的字典
```

函数修改了传入的 `ai_parsed` 字典，这是一个**隐式副作用**。调用方（pipeline.py L318-321）依赖这个副作用来更新 `ai_parsed`，但函数签名并未体现这一点。

**建议：** 要么在文档字符串中明确说明副作用，要么返回修改后的字典由调用方合并。

## 四、性能问题

### 🟡 中等

**1. flight_delay/pipeline.py — 飞常准候选航班串行查询（L382-464）**

```python
for candidate_fn, candidate_dep, candidate_arr, candidate_date in cf_candidates:
    ...
    one_result = await skill.lookup_status(...)  # 串行 await
```

多个候选航班逐个串行查询，每个查询都是一次网络IO。如果候选列表有 3 个航班，总耗时 = 3 × 单次查询耗时。

**建议：** 使用 `asyncio.gather()` 并行查询：
```python
tasks = [skill.lookup_status(...) for ... in cf_candidates]
results = await asyncio.gather(*tasks, return_exceptions=True)
```

**2. delay_calc.py — 全量时间解析（L94-137）**

`_compute_delay_minutes` 在函数开头解析了所有时间字段（chain、schedule、alternate、actual），即使某些口径不需要全部数据。例如口径1（chain→actual）不需要 alternate 的时间。

**建议：** 改为惰性解析，仅在需要时计算。或至少将解析逻辑拆分为独立函数。

**3. baggage_delay/pipeline.py — 视觉合并循环中的重复字符串转换（L112-134）**

```python
for key in (...22个字段...):
    vision_val = vision_extract.get(key)
    parsed_val = ai_parsed.get(key)
    if vision_val is not None and str(vision_val).lower() not in ("unknown", "", "[]"):
        ai_parsed[key] = vision_val
```

22 个字段逐个处理，每个都做 `str().lower()` 转换。虽然单次开销很小，但可以抽取公共函数简化。

## 五、可维护性

### 🔴 严重

**1. 两个 pipeline 均超过 500 行限制**

| 文件 | 实际行数 | 超出 |
|------|---------|------|
| `baggage_delay/pipeline.py` | 523 | +23 (4.6%) |
| `flight_delay/pipeline.py` | 729 | +229 (45.8%) |

项目规范（CLAUDE.md）明确要求 pipeline ≤ 500 行。`flight_delay/pipeline.py` 超出近 46%，严重违反规范。

**建议：**
- `baggage_delay/pipeline.py`：将视觉合并逻辑（L112-246, ~135行）提取到 `stages/vision_merge.py`
- `flight_delay/pipeline.py`：将以下三段分别提取：
  - 视觉合并（L139-329, ~190行）→ `stages/vision_merge.py`
  - 飞常准查询（L331-467, ~136行）→ `stages/aviation_lookup.py`
  - 替代航班查询（L469-608, ~140行）→ `stages/alt_flight_lookup.py`

### 🟡 中等

**2. hardcheck.py — `_run_hardcheck` 函数过长（~516行）**

单个函数承载了 10+ 种不同的校验逻辑，违反单一职责原则。

**建议：** 拆分为独立函数：
- `_check_airport_info()` — 机场信息解析
- `_check_war_risk()` — 战争风险
- `_check_policy_window()` — 保单有效期
- `_check_missed_connection()` — 中转接驳
- `_check_required_materials()` — 必备材料

**3. delay_calc.py — 内部函数应提升为模块级**

`_try_parse_utc`、`_try_parse_local`、`_resolve_iana` 三个内部函数定义在 `_compute_delay_minutes` 内部，但它们不依赖闭包变量（除了 `_dep_iana`/`_arr_iana`），可以提升为模块级函数，便于单独测试和复用。

**4. 多处重复定义 `_iata` 和 `_parse_date_str`**

| 函数 | 定义位置 |
|------|---------|
| `_iata()` | hardcheck.py L106, L189 |
| `_parse_date_str()` / `_parse_date_any()` | hardcheck.py L58, L199 |

同一文件中两个函数各自定义了几乎相同的辅助函数。

**建议：** 提取到 `utils.py` 统一复用。

**5. 变量命名不够表意（delay_calc.py）**

```python
a: Optional[int] = None  # 口径1-起飞延误
b: Optional[int] = None  # 口径1-到达延误
c: Optional[int] = None  # 口径2-起飞延误
d: Optional[int] = None  # 口径2-到达延误
e: Optional[int] = None  # 口径3-起飞延误
f: Optional[int] = None  # 口径3-到达延误
```

单字母变量名 `a`~`f` 需要对照注释才能理解含义。

**建议：** 使用描述性名称：
```python
chain_dep_delay: Optional[int] = None
chain_arr_delay: Optional[int] = None
alt_dep_delay: Optional[int] = None
alt_arr_delay: Optional[int] = None
actual_dep_delay: Optional[int] = None
actual_arr_delay: Optional[int] = None
```

### 🔵 建议

**6. 两个 pipeline 结构不一致**

| 特性 | baggage_delay | flight_delay |
|------|-------------|-------------|
| 重复理赔检测 | ❌ 无 | ✅ stage0_duplicate |
| 硬校验阶段 | ❌ 分散在各处 | ✅ stage_hardcheck 集中 |
| 赔付预计算 | ❌ 最后计算 | ✅ stage10 提前计算 |
| 规则兜底后处理 | ❌ 无 | ✅ postprocess |
| 错误处理 | 内联 try/except | build_stage_error_return |

两个险种的 pipeline 架构差异较大，`flight_delay` 明显更成熟（有重复检测、硬校验集合、后处理），`baggage_delay` 缺少这些阶段。

**建议：** 将 `flight_delay` 的成熟模式（重复检测、硬校验、后处理）回补到 `baggage_delay`。

**7. 魔数散落各处**

| 魔数 | 位置 | 含义 |
|------|------|------|
| `6` | baggage_delay L432 | 行李延误赔付门槛（小时） |
| `5 * 60` | delay_calc.py L298 | 航班延误默认门槛（分钟） |
| `3` | hardcheck.py L141 | 欺诈检测天数阈值 |
| `1.2` | delay_calc.py L243 | 联程改签判定系数 |
| `4000` | utils.py L18 | 条款截断长度 |
| `1000` | baggage_delay L53 | 条款摘要截断长度 |

**建议：** 统一定义为模块级常量：
```python
BAGGAGE_DELAY_THRESHOLD_HOURS = 6
FLIGHT_DELAY_DEFAULT_THRESHOLD_MINUTES = 300
FRAUD_SUSPECT_DAYS_THRESHOLD = 3
CONNECTING_REBOOKING_RATIO = 1.2
POLICY_EXCERPT_MAX_CHARS = 4000
```

**8. 延迟导入（late import）**

```python
# baggage_delay/pipeline.py L496
from app.modules.baggage_delay.stages.utils import _safe_float

# flight_delay/pipeline.py L332
from app.skills.flight_lookup import get_flight_lookup_skill
```

函数中间的 import 语句降低可读性，且 `_safe_float` 实际上在文件顶部已通过 `from app.modules.baggage_delay.stages.utils import (...)` 导入其他函数，应一并导入。
